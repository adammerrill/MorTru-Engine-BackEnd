End Goal — Loan Scenario Analysis API

What the system must ultimately produce, and what it must accept as input.

Scope Boundary (User-Defined) In scope: Front-end DTI (housing expense / gross monthly income). The engine computes PITIA and compares it to the borrower's gross monthly income.
Out of scope: Back-end DTI, existing debts, credit card payments, student loans. These require a licensed loan officer and are deliberately excluded.

The API Input

The endpoint accepts two input formats — native JSON or MISMO 3.4 XML. Both routes produce identical analysis output.

Critical Gaps

BorrowerProfile — the core input type does not exist
Critical

The entire analysis pipeline receives borrower data, but there is no BorrowerProfile struct anywhere in the codebase. Every downstream epic (eligibility, pricing, scenarios, orchestrator) needs this type. It must be defined in types so all crates can depend on it without circular references.

// Needs to be added to crates/types/src/borrower_profile.rs
pub struct BorrowerProfile {
    // ── Income (front-end DTI only) ───────────────────────────────────────
    /// Combined gross monthly income of all borrowers on the note.
    /// This is what drives front-end DTI. Back-end DTI is out of scope.
    pub gross_monthly_income: Cents,

    // ── Credit ────────────────────────────────────────────────────────────
    /// Representative credit score per industry convention:
    ///   Single borrower: middle of 3 bureau scores
    ///   Co-borrower: lower of the two borrowers' middle scores
    pub representative_credit_score: CreditScore,

    // ── Down payment ──────────────────────────────────────────────────────
    pub down_payment: Cents,
    pub down_payment_source: DownPaymentSource, // OwnFunds | Gift | Bridge

    // ── Program eligibility flags ─────────────────────────────────────────
    pub occupancy: Occupancy,
    pub loan_purpose: LoanPurpose,
    pub is_first_time_buyer: bool,
    pub veteran_status: Option<VeteranStatus>,  // None = not a veteran

    // ── USDA-specific (household, not just borrowers) ─────────────────────
    /// Number of people (including non-borrowers) who will live in the home.
    /// USDA uses household income from ALL adults, not just borrowers.
    pub household_size: Option<u8>,
    /// Annual income of all adults in the household (USDA only).
    pub annual_household_income: Option<Cents>,

    // ── Co-borrower raw scores (for representative score calculation) ──────
    pub coborrower_credit_scores: Vec<CreditScore>,
    pub coborrower_monthly_incomes: Vec<Cents>,
}
VeteranStatus enum — missing
Critical

VA funding fee differs significantly by veteran status. Without this enum, VA scenarios cannot compute the funding fee, which is the largest cost difference between first-use and subsequent-use VA loans.

pub enum VeteranStatus {
    /// First-time VA loan use, no disability. Funding fee: 2.15% (3.5% if >10% down already paid → 1.5%)
    FirstUse,
    /// Subsequent use of VA benefit. Funding fee: 3.3%.
    SubsequentUse,
    /// Disability-exempt. Funding fee: 0% (waived entirely).
    DisabilityExempt,
    /// National Guard / Reserves - different funding fee than active duty
    NationalGuardFirstUse,
    NationalGuardSubsequent,
}
LoanScenarioResult + PitiaBreakdown output types — missing
Critical

The API response needs typed output structs. Currently there is no LoanScenarioResult, PitiaBreakdown, QualificationStatus, or ReasonCode type. Without these, no downstream crate can return a structured analysis result.

High

Three Additions Needed

monthly_tax_cents() helper — trivial but important
High

tax_annual_cents is in PropertyEnriched but there is no monthly_tax_cents() helper. The PITIA calculation needs monthly tax, and dividing by 12 with correct rounding (round-up to be conservative for borrower benefit) should be a method, not inline arithmetic repeated at every call site.

// Add to property/listing.rs
impl PropertyReso {
    /// Monthly property tax = TaxAnnualAmount / 12, rounded up to the nearest cent.
    /// Conservative rounding protects borrower from underestimating payment.
    pub fn monthly_tax_cents(&self) -> Option<Cents> {
        self.tax_annual_cents.map(|annual| {
            Cents((annual.0 + 11) / 12)  // ceiling division
        })
    }
}
HOI monthly estimate — property data almost never includes HOI
High

RESO data does not include homeowner's insurance premium. For PITIA calculation the engine needs an HOI estimate. RESO properties have the property state and value — an estimator can use state-average rates to produce a reasonable monthly amount. Without this, the engine cannot compute a complete PITIA or front-end DTI.

Estimation method: Annual HOI ≈ property_value × state_rate. For example, Texas averages ~0.17% per year. $459,000 × 0.0017 / 12 = $65/month. This is stored in ref_data as a state-level rate table (Task 4.22, new).

Note: RESO does not have an InsurancePremiumAnnual field. The estimate is always required unless the borrower provides actual HOI quote data.

Flood insurance monthly estimate — when required, amount is unknown
Medium

When flood_insurance_required = true (FloodZone AE/VE/etc.), the engine knows flood insurance is needed but has no amount to include in PITIA. A state or zone-based minimum estimate is needed — FEMA's minimum NFIP premium for a residential structure in a SFHA zone is approximately $700–$1,200/year. A conservative floor estimate of $1,200/year ($100/month) should be used when the actual premium is unknown.

HOI estimation by state — NOT IN THE PLAN
High

RESO data never includes HOI (homeowner's insurance) premiums. To compute PITIA, the engine must estimate HOI. The estimation method: purchase_price × state_annual_rate / 12. This requires a ref_data table of annual HOI rate by state (e.g. TX = 0.17%, CA = 0.08%, FL = 0.21%). Without this, PITIA and front-end DTI cannot be calculated for any property.

FHA condo project approval — NOT IN THE PLAN
Medium

Not all condominiums are FHA-approved. For a condo to receive FHA financing, the project must be on the HUD Approved Condominiums list (or use single-unit approval for certain conditions). Without checking HUD's condo approval database, the engine will incorrectly mark some condos as FHA-eligible. The HUD condo API is queryable at https://www.hud.gov/program_offices/housing/sfh/apprlenderinstruct.

Recommended approach: In the analysis output, flag all condo FHA scenarios as "pending condo project approval verification" rather than fully qualified. The eligibility engine should note this with reason code FHA_CONDO_PROJECT_APPROVAL_REQUIRED.

BorrowerProfile — Complete Design

New type needed in Epic 1. This is the single most important missing piece — every downstream epic needs this type.

// crates/types/src/borrower_profile.rs

/// How the borrower's down payment funds were sourced.
/// Source affects gift documentation requirements and program eligibility.
#[derive(Copy, Clone, Debug, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum DownPaymentSource {
    OwnFunds,      // savings, investments, sale of asset — no restriction
    Gift,          // gift from family member — documentation required
    Bridge,        // equity bridge from home being sold — affects timing
    DownPaymentAssistance, // DPA grant/loan from HFA or nonprofit
}

/// The borrower's VA entitlement and funding fee tier.
#[derive(Copy, Clone, Debug, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum VeteranStatus {
    FirstUse,                    // 2.15% fee (standard purchase); 0% if disabled
    SubsequentUse,               // 3.30% fee
    DisabilityExempt,            // 0% fee — always preferred
    NationalGuardReservesFirst,  // 2.40% fee (Guard/Reserves)
    NationalGuardReservesSubseq, // 3.30% fee
}

/// Borrower's employment / income type.
/// Affects income documentation requirements (not income amount or DTI calc).
#[derive(Copy, Clone, Debug, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub enum EmploymentType {
    W2FullTime,    // standard W-2 employment — simplest documentation
    W2PartTime,    // part-time; must show 2yr history to count full amount
    SelfEmployed,  // 2yr tax returns required; net income used
    Commission,    // 2yr average if variable
    Retired,       // pension/SS income — stable; good for programs
    Military,      // base pay + BAH eligible
}

/// The complete borrower input for the analysis engine.
pub struct BorrowerProfile {
    // ── Income (front-end DTI only — back-end is out of scope) ────────────────
    /// Gross monthly qualifying income for all borrowers combined.
    /// This is the number used for front-end DTI.
    /// W-2: gross salary / 12
    /// Self-employed: 2-year average net income / 12
    pub gross_monthly_income: Cents,
    pub employment_type: EmploymentType,

    // ── Credit ────────────────────────────────────────────────────────────────
    /// Representative credit score per agency convention:
    ///   Single borrower: middle score of 3 bureau pulls
    ///   Co-borrowers: lower of each borrower's middle score
    /// Engine uses this for all eligibility and pricing tier lookups.
    pub representative_credit_score: CreditScore,

    // ── Transaction ───────────────────────────────────────────────────────────
    pub occupancy: Occupancy,
    pub loan_purpose: LoanPurpose,

    // ── Down payment ──────────────────────────────────────────────────────────
    pub down_payment: Cents,
    pub down_payment_source: DownPaymentSource,

    // ── Program eligibility flags ─────────────────────────────────────────────
    /// First-time homebuyer: required for HomeOne; expands options for HomeReady/HP.
    pub is_first_time_buyer: bool,
    /// VA entitlement details. None = borrower is not a veteran.
    pub veteran_status: Option<VeteranStatus>,
    /// Existing VA loan not yet paid off → reduced/impaired entitlement.
    /// Affects whether loan limit applies to VA loans.
    pub has_existing_va_loan: bool,

    // ── USDA-specific (different income definition) ───────────────────────────
    /// Total people who will live in the home (all ages).
    /// Used to look up USDA SFGH income limit for the right family size.
    /// None = USDA income check deferred / not applicable.
    pub household_size: Option<u8>,
    /// Combined annual income of ALL adults (18+) in the home, including
    /// non-borrowers. This is the USDA-specific income figure.
    /// Different from gross_monthly_income × 12 when non-borrower adults exist.
    pub annual_household_income: Option<Cents>,

    // ── Co-borrower details ───────────────────────────────────────────────────
    /// For the analysis, co-borrower incomes are already rolled into
    /// gross_monthly_income above. These are kept separately for:
    /// 1) Representative credit score calculation (lower of middle scores)
    /// 2) Verification: combined income ≥ gross_monthly_income
    pub coborrower_credit_scores: Vec<CreditScore>, // empty if no co-borrower
}

Front-End DTI — End-to-End Path

What Front-End DTI Means in This Engine

Front-End DTI = PITIA / Gross Monthly Income

Where PITIA = Principal + Interest + Taxes + Insurance + Association (HOA)

Scope BoundaryThe engine calculates front-end DTI only. Back-end DTI (which adds all monthly debt obligations to housing cost) requires knowledge of credit card minimums, student loans, auto loans, and other debts. These must be provided by a licensed loan officer and are out of scope. The analysis output clearly labels all DTI figures as "front-end only."

T — TaxesPropertyEnriched.tax_annual_centsAnnual / 12 (ceiling)
I — Insurance (HOI)PropertyEnriched or estimateFrom RESO data or state rate × value / 12

DTI Calculation Example — Kyle TX SFR

Inputs:
  Purchase price:    $459,000
  Down payment:      $16,065 (3.5%)
  Base loan:         $442,935
  UFMIP (FHA 1.75%): $7,751 (financed)
  Total loan:        $450,686
  Rate:              6.875% (illustrative)
  Term:              360 months
  Gross income:      $8,500/mo (single borrower)

PITIA breakdown:
  P + I:   $2,959/mo  (amortization: $450,686 × 6.875% / 12, 360 months)
  T:       $   742/mo  ($8,900/yr ÷ 12, ceiling)
  I (HOI): $   215/mo  (state estimate: $459,000 × 0.0056% ≈ TX average)
  I (MIP): $   313/mo  (FHA: 0.85% × $442,935 / 12 = $313.41)
  A (HOA): $    75/mo  (from PropertyEnriched)
  ─────────────────────
  PITIA:   $4,304/mo

Front-end DTI:  $4,304 / $8,500 = 50.6%  ← EXCEEDS FHA 31% limit → NOT QUALIFIED

With co-borrower adding $3,500/mo:
  Combined income: $12,000/mo
  Front-end DTI:  $4,304 / $12,000 = 35.9%  ← still exceeds FHA 31% limit

Conclusion: $459,000 purchase with 3.5% down at 6.875% does not qualify
on front-end DTI for FHA at $8,500–$12,000/mo income.
The engine surfaces this with REASON_CODE: FHA_FRONT_END_DTI_EXCEEDED.

PITIA Breakdown — Type Design

// crates/types/src/pitia.rs — new type needed

/// Complete housing payment breakdown for one loan scenario.
/// All amounts are monthly in Cents. P+I is from the amortization engine.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PitiaBreakdown {
    // ── P + I ─────────────────────────────────────────────────────────────
    pub monthly_principal_and_interest: Cents,

    // ── T ─────────────────────────────────────────────────────────────────
    pub monthly_property_tax: Cents,
    /// True if this is an estimate (tax not in property data — computed
    /// from assessed value × estimated rate).
    pub property_tax_estimated: bool,

    // ── I (standard HOI) ──────────────────────────────────────────────────
    pub monthly_homeowners_insurance: Cents,
    /// True if HOI is a state-average estimate (no actual quote provided).
    pub hoi_estimated: bool,

    // ── I (program MI) ────────────────────────────────────────────────────
    /// Monthly mortgage insurance (FHA MIP, USDA annual fee, or Conv PMI).
    /// Zero for VA (no monthly MI).
    pub monthly_mortgage_insurance: Cents,
    pub mortgage_insurance_type: MortgageInsuranceType,
    /// LTV at which monthly MI cancels. None = life of loan (FHA >90% LTV).
    pub mi_cancels_at_ltv: Option<LtvBasisPoints>,

    // ── I (flood — only when in SFHA) ─────────────────────────────────────
    /// Monthly flood insurance. Zero when not in SFHA zone.
    pub monthly_flood_insurance: Cents,
    pub flood_insurance_estimated: bool,

    // ── A ─────────────────────────────────────────────────────────────────
    /// Monthly HOA dues. Zero if no HOA.
    pub monthly_hoa: Cents,

    // ── Totals ────────────────────────────────────────────────────────────
    pub total_pitia: Cents,
}

impl PitiaBreakdown {
    /// Front-end DTI as basis points. Caller provides gross monthly income.
    pub fn front_end_dti(&self, gross_monthly_income: Cents) -> DtiBasisPoints {
        if gross_monthly_income.0 <= 0 { return DtiBasisPoints::new(u32::MAX); }
        let bps = (i64::from(self.total_pitia.0) * 10_000
                   / i64::from(gross_monthly_income.0)) as u32;
        DtiBasisPoints::new(bps)
    }
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
pub enum MortgageInsuranceType {
    FhaMip,          // FHA mortgage insurance premium
    ConventionalPmi, // private mortgage insurance
    UsdaAnnualFee,   // USDA annual guarantee fee (0.35%)
    None,            // VA, or conventional ≥20% down
}

MISMO 3.4 Full Coverage — Output Path

Current StateEpic 2 built MISMO 3.4 input parsing (MISMO XML → ParsedDeal). "Full MISMO coverage" requires the output direction too: LoanAnalysisResult → MISMO 3.4 XML. This is currently undesigned and unbuilt.

What MISMO Output Enables

  • LOS Integration: All major Loan Origination Software (Encompass, BytePro, Calyx) can import MISMO 3.4 deal set format. Outputting MISMO lets borrowers take their analysis directly to a lender's LOS.
  • AUS Submission: Fannie Mae's Desktop Underwriter (DU) and Freddie Mac's Loan Product Advisor (LPA) both accept MISMO 3.4 loan data for automated underwriting. Generating MISMO output enables AUS submission.
  • Audit Trail: CFPB examination prefers MISMO as the industry-standard representation of loan data. MISMO XML is self-documenting and schema-validated.
  • Downstream Processing: Title companies, appraisers, and MI providers all accept MISMO 3.4.

MISMO Output Scope (Task 2.13, new)

For each qualified scenario, produce a MISMO 3.4 MESSAGE containing:

  • DEAL/PARTIES/PARTY/BORROWER_DETAIL — credit score, income, occupancy
  • DEAL/COLLATERALS/COLLATERAL/SUBJECT_PROPERTY — address, property type, value
  • DEAL/LOANS/LOAN/MORTGAGE_TERMS — loan amount, rate, term, program
  • DEAL/LOANS/LOAN/QUALIFYING_RATE_PERCENT — rate used for DTI
  • DEAL/LOANS/LOAN/PROPOSED_HOUSING_EXPENSE — P+I, taxes, insurance, HOA (PITIA)
  • DEAL/LOANS/LOAN/QUALIFY_INCOME — gross monthly income
  • DEAL/LOANS/LOAN/MI_DATA — if applicable

The output does not need to include liabilities, full borrower PII, or any data the engine doesn't have. A partial MISMO document that covers the loan/property/borrower financial profile is sufficient and standard practice for point-of-sale tools.
