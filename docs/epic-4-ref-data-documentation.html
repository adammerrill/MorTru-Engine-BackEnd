Task 4.2 — JsonFileStore (5 tests) ✅

What It Does

Full implementation of all 10 RefDataStore trait methods for JsonFileStore. The store reads JSON files from data_dir/ on demand — no database required for dev and CI.

File Discovery — Automatic Year Fallback

Every versioned dataset uses the naming pattern {prefix}_{year}.json. When a year is requested, the store scans data_dir/ and returns data from the most recent year ≤ the requested year. Requesting 2030 data returns 2025 data with no code changes. Adding a fha_limits_2026.json file automatically makes it available.

// Year fallback example:
store.fha_loan_limits("48209", 2030)
// Scans data_dir/ → finds fha_limits_2025.json → returns 2025 data

Static datasets (no year in filename)

USDA rural eligibility and MFH by tract use {name}.json without a year suffix — these datasets update on a 10-year Census cycle, not annually.

Task 4.15 — Conventional MI Coverage + LLPA

Required MI coverage percentages by LTV (per FNMA/FHLMC guidelines) and LLPA price adjustments. LLPAs are the credit-score × LTV grid that determine the rate premium above par.

Phase 5 — Lender and Rate Data (Tasks 4.16–4.18)

Task 4.16 — LenderProfile + LenderOverlays

Per-lender restrictions that tighten (but never loosen) agency guidelines. A lender might require a minimum 640 FICO for FHA even though agencies allow 580. Overlays are applied after program eligibility rules and can only disqualify — never qualify — a scenario.

pub struct LenderOverlay {
    pub program: ProgramCode,
    pub min_credit_score: Option<u16>,       // tighten agency min
    pub max_ltv_bps: Option<u32>,            // tighten agency max
    pub front_end_dti_max_bps: Option<u32>,  // tighten agency DTI
    pub ineligible_property_types: Vec<PropertyType>,
    pub ineligible_states: Vec<StateCode>,
}

Task 4.17 — MI Provider + MiRateCard

Monthly MI rate grids by LTV bracket, credit score tier, coverage level, and loan term. Major providers: MGIC, Essent, Radian, Arch, National MI. Rates are typically weekly or monthly updates.

Task 4.18 — RateSheet (volatile intraday)

Par interest rates and YSP (Yield Spread Premium) curves for each loan product and lock period. The most volatile data in the system — updates multiple times per day. Requires sub-hour versioning. The rate sheet is the final input needed to compute a complete monthly P+I payment and therefore a complete PITIA.

Rate Sheet DependencyWithout Task 4.18, the engine can evaluate eligibility (pass/fail) but cannot compute the actual monthly payment or front-end DTI at a specific rate. The analysis can still be produced for a user-provided rate, but automatic rate selection requires a live rate sheet.

Phase 6 — Integration Gate (Tasks 4.19, 4.23, 4.20)

Task 4.19 — GeoEligibility Unified Query (full implementation)

The geo_eligibility() method is already partially implemented in JsonFileStore — it assembles FHA limits, GSE limits, and USDA data. Task 4.19 completes it: adds proper error handling for missing tract data, verifies the AMI integration, adds the SqliteStore implementation, and writes comprehensive integration tests covering the single-call pattern.

Task 4.23 — FHA Condo Project Approval (Audit)

Not all condominiums are FHA-approved. The HUD Approved Condominiums list is queryable via API. For the engine, all FHA condo scenarios are flagged FHA_CONDO_PROJECT_APPROVAL_REQUIRED unless the project appears in the approved list. This prevents the engine from incorrectly qualifying condo FHA scenarios.

Program Rules Deep Dive

Evaluation Order (every scenario)

  1. Property type ineligibility — mobile home, investment property → fail before any data lookup
  2. Occupancy check — USDA/VA/HomeOne require primary residence
  3. VA entitlement — if program requires VA, borrower must have entitlement
  4. First-time buyer — HomeOne requires at least one FTIB on the note
  5. Credit score minimum — program_rules.credit_score_eligible()
  6. LTV maximum — program_rules.ltv_eligible()
  7. Front-end DTI — program_rules.dti_eligible()
  8. Geographic: FHA/GSE loan limits — geo.fha_loan_within_limit() / geo.gse_loan_status()
  9. Geographic: USDA rural — geo.usda_sfh_eligible
  10. Geographic: USDA income — geo.usda_income_eligible()
  11. AMI income check — HomeReady/HP: geo.hp_income_eligible()
  12. Lender overlays — tighten (never loosen) any of the above
VA front-end DTIVA stores front_end_dti_max_bps = 9999. This means Step 7 always passes for VA, which is correct — VA uses residual income, not front-end DTI. The analysis output notes this explicitly: "VA uses residual income — DTI check not applicable. A loan officer will verify residual income separately."

Lender Overlay Priority

The engine applies rules in this precedence order (most restrictive wins):

  1. Agency/GSE guidelines (program_rules.json)
  2. Investor guidelines (FNMA/FHLMC selling guide deviations)
  3. Lender overlays (LenderOverlay — Task 4.16)
  4. MI provider requirements — some MI companies have their own credit overlays
