Key Design Decisions

VA DTI = 9999: VA does not use front-end DTI. Storing 9999 bps (99.99%) makes the DTI check always pass for VA, which is correct — a loan officer verifies residual income separately. This avoids special-casing VA in every caller.

FHA two-tier credit: credit_score_eligible(score, down_payment_bps) handles the 580/500 split automatically. Callers pass the down payment and get the right minimum without knowing about the tier logic.

HOI ceiling division: monthly_estimate() uses (annual + 11) / 12 — ceiling division ensures we never underestimate the monthly payment. Conservative protects the borrower from qualifying on a payment that's actually too low.

Year fallback: Requesting year 2030 returns 2025 data. Crate never needs code changes when a new year's data file is added — it just becomes the new best match.

Why SqliteStore?

The production deployment uses PostgreSQL. The SqliteStore serves a different purpose: it gives integration tests a real relational database without needing a running PostgreSQL server in CI. When a test creates a SqliteStore, it creates an in-process, in-memory database, runs the schema migrations, loads the JSON seed data, and is ready to use — all in milliseconds.

Both stores implement the same RefDataStore trait, so every test written against one store verifies behavior for both. The SqliteStore is also used in development when a developer doesn't have PostgreSQL running locally.

Task 4.4 — SqliteStore

Implements all 10 RefDataStore methods using in-process SQLite via sqlx.

Design

// crates/ref_data/src/store/sqlite.rs

/// SQLite-backed store — used for integration tests and local dev without PostgreSQL.
///
/// # Usage in tests
///
/// ```rust
/// let store = SqliteStore::new_test_store().await.unwrap();
/// let limits = store.fha_loan_limits("48209", 2025).unwrap();
/// ```
pub struct SqliteStore {
    pool: sqlx::SqlitePool,
}

impl SqliteStore {
    /// Create an in-memory SQLite database, apply schema, and seed from JSON files.
    /// This is the test-facing constructor. Production uses `from_pool()`.
    pub async fn new_test_store() -> RefDataResult<Self>;

    /// Connect to an existing SQLite file.
    pub async fn from_url(url: &str) -> RefDataResult<Self>;

    /// Use a caller-provided pool (production: PostgreSQL behind a compatible wrapper).
    pub fn from_pool(pool: sqlx::SqlitePool) -> Self;

    /// Apply SQLite-compatible schema (inline CREATE TABLE statements).
    async fn init_schema(pool: &sqlx::SqlitePool) -> RefDataResult<()>;

    /// Load all JSON data files into the database.
    async fn seed_from_json(pool: &sqlx::SqlitePool) -> RefDataResult<()>;
}

Test Strategy

The SqliteStore test file mirrors the JsonFileStore tests, calling the same store methods with the same expected values. This verifies that both backends produce identical results for all queries. Key tests beyond the mirroring:

  • test_sqlite_store_from_in_memory_db() — store creates successfully
  • test_sqlite_schema_all_tables_exist() — all 9 tables created
  • test_sqlite_seed_data_loaded() — row counts match JSON file record counts
  • test_sqlite_year_lookup_uses_most_recent() — SQL ORDER BY / LIMIT logic
  • test_sqlite_fha_results_match_json_store() — both return same Cents values
  • test_sqlite_program_rules_match_json_store() — same DTI/LTV/credit rules
  • test_sqlite_geo_eligibility_matches_json_store() — same GeoEligibility result

Task 4.11 — CBSA/MSA Crosswalk

Maps county FIPS → CBSA (Core Based Statistical Area) code and designation (Metropolitan / Micropolitan / Rural). Source: Census OMB delineation files, updated every 3–5 years.

// New method on RefDataStore trait
fn cbsa_for_county(&self, fips_code: &str) -> RefDataResult<Option<CbsaEntry>>;

pub struct CbsaEntry {
    pub fips_code: String,
    pub cbsa_code: Option<CbsaCode>,    // None = rural (no CBSA)
    pub cbsa_name: Option<String>,
    pub designation: CbsaDesignation,   // Metropolitan | Micropolitan | Rural
    pub is_metro: bool,                 // true for Metropolitan only
}

Used downstream by the USDA eligibility check: some counties that appear urban by FIPS can still have USDA-eligible tracts if they're classified as Micropolitan rather than Metropolitan. Also used for MSA-level income adjustments.

Task 4.12 — Texas HOI Premiums by ZIP

Texas Department of Insurance publishes annual homeowner's insurance premium data by ZIP code. More precise than the state-average 56 bps from Task 4.22.

// New method on RefDataStore trait (optional — falls through to state_hoi_rate)
fn zip_hoi_rate(&self, zip5: &str, year: u16) -> RefDataResult<Option<ZipHoiRate>>;

pub struct ZipHoiRate {
    pub zip5: String,
    pub state_abbr: String,             // always "TX" for this dataset
    pub annual_rate_bps: u16,
    pub median_premium_cents: Cents,    // TDOI median annual premium
    pub sample_size: u32,
    pub effective_year: u16,
}

Fall-through lookup pattern (used by the analysis engine):

fn hoi_monthly(store, zip, state, value) -> Cents {
    // 1. Try ZIP-level (Task 4.12) — only TX for now
    if let Some(r) = store.zip_hoi_rate(zip, year)? {
        return r.monthly_estimate(value);
    }
    // 2. Fall through to state average (Task 4.22)
    let r = store.state_hoi_rate(state, year)?;
    r.monthly_estimate(value)
    // 3. National fallback (NATIONAL_FALLBACK_RATE_BPS) used if state also missing
}
