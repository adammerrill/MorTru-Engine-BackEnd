Key Design Decisions

Flat struct — no nested sub-structs

The RESO Web API returns a flat JSON object. PropertyReso mirrors this exactly — one flat struct with 235 Option<T> fields and #[serde(rename = "PascalCase")] on every field. Nested sub-structs would require custom deserialization and add indirection at every call site.

No deny_unknown_fields

RESO updates its Data Dictionary periodically. New fields added in DD 2.1 or DD 3.0 are silently ignored rather than causing parse failures. This is intentional — it avoids breaking changes when MLS boards add fields.

All fields are Option<T>

RESO Web API responses omit fields with no value. Treating a missing field as an error would reject a large fraction of real-world listings. enrich() enforces required fields at the boundary instead.

PropertyEnriched.raw preserves all 235 fields

Box<PropertyReso> is moved into raw at the end of enrich() — zero copies. Any field not promoted into PropertyEnriched is accessible via enriched.raw.field_name.

Acres → sq ft via Decimal

lot_size_sqft() converts acres using Decimal::from(43_560u32), not f64. This avoids the floating-point drift that would cause 0.2 × 43560.0 to produce 8711.999... instead of exactly 8712.

FCC Census Geocoder Client

API

GET https://geo.fcc.gov/api/census/block/find
    ?latitude={lat}&longitude={lon}&censusYear=2020&format=json

Response Shape

{
  "status": "OK",
  "County": { "FIPS": "48209", "name": "Hays" },
  "State":  { "FIPS": "48",    "code": "TX",   "name": "Texas" },
  "Block":  { "FIPS": "482090109053009" }
}

Block.FIPS is 15 digits: state(2) + county(3) + tract(6) + block(4).
Census tract GEOID = first 11 digits = "48209010905".

Design Decisions

Flat struct vs nested sub-structs

Nested sub-structs require custom Deserialize implementations because RESO Web API returns a flat JSON object. A flat struct with #[serde(rename)] on every field handles the wire format directly with zero custom code. Navigation is via method modules (address.rs, physical.rs, etc.) not struct hierarchy.

No deny_unknown_fields

RESO Data Dictionary releases (2.0 → 2.1 → 3.0) add new fields. MLS boards add local extensions. deny_unknown_fields would cause parse failures on new fields, breaking the pipeline every time RESO updates. Unknown fields are silently dropped — no data is lost because all fields we consume are already in the struct.

HOA normalization in hoa_monthly_cents()

PITIA (Principal + Interest + Tax + Insurance + Association) calculation requires a monthly recurring figure. MLS boards use all five RESO frequencies inconsistently. Normalizing at the PropertyReso level means every downstream consumer gets a consistent monthly Cents value without repeating the frequency logic. OneTime fees (transfer/initiation fees) return None — they are not recurring expenses.

Year built: Err vs graceful None

year_built() returns Err for values outside 1800–2040 — useful for strict validation contexts. enrich() uses year_built().unwrap_or(None) to absorb the error gracefully, because an out-of-range year is a data quality issue that should not prevent the property from being analyzed.

Null island coordinate rejection

Some MLS data export pipelines geocode failed lookups to (0.0, 0.0) instead of leaving the field null. validated_coordinates() explicitly rejects this sentinel value as InvalidCoordinate.

lot_size_sqft() via Decimal arithmetic

The conversion factor is exactly 43,560 sq ft per acre. Using f64 arithmetic, 0.2 × 43560.0 can produce 8711.999999999998 which truncates to 8,711 instead of 8,712. Using rust_decimal, the multiplication is exact.

FCC client: parse function separated from HTTP

All test coverage of the FCC response format is against parse_fcc_response(json) — a pure function that takes a string and returns a FipsResolution. This gives 100% coverage of the parsing logic without any HTTP infrastructure. The stub FccClient::resolve() is replaced with a real async call in Epic 5.
