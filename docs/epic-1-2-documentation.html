The MorTru Engine is a high-precision mortgage pricing and scenario analysis engine. It produces analysis across all eligible loan programs for a given scenario, pricing against multiple lenders and MI providers simultaneously.

BasisPoints
newtype u32 · scale depends on context
Interest rates use ×1000 scale (6.375% = 6375). Fee rates and MI rates use ×100 scale (1.75% = 175). Context determines which scale applies.

DtiBasisPoints
newtype u32 · 1 unit = 0.01% DTI
Debt-to-income ratios. Warns above TYPICAL_MAX (6000 = 60%). Used for both front-end (housing) and back-end (total) DTI.

Architecture Decisions

Key decisions made during Epics 1–2 and their rationale

Precision architecture

No f64 for storage — ever
All monetary values are Cents (i64). All rates are BasisPoints (u32). All intermediate calculations use rust_decimal::Decimal with half-up rounding. The only f64 in the engine is in Newton-Raphson APR convergence iterations — intermediate values only, never stored. This eliminates an entire class of floating-point rounding errors that historically cause mortgage calculation discrepancies.
BasisPoints dual-scale convention
The BasisPoints type uses ×1000 scale for interest rates (1 unit = 0.001%, so 6.375% = 6375) and ×100 scale for fee/MI rates (1 unit = 0.01%, so 1.75% UFMIP = 175). This follows industry convention: interest rates are quoted to 3 decimal places while fee/MI rates are quoted to 2. The scale is determined by context, not by the type itself.

MISMO schema layer

Parse-only crate — no business logic in mismo
The mismo crate converts XML ↔ typed Rust structs. It validates format and range but applies zero business rules. This separation allows the schema layer to be updated (e.g., new MISMO fields) without touching any calculation logic in downstream crates.
DU "Approve/Eligible" slash-form bug fix
Desktop Underwriter (DU) actually emits "Approve/Eligible" (with slash) in live MISMO XML. The MISMO specification document only shows "ApproveEligible" (camelCase). The AusRecommendation::try_from_str parser now accepts both forms. Any production MISMO import from DU would silently fail to parse its recommendation without this fix.

Fee rules architecture

fee_rules.json — externally maintainable, no redeployment
All fee type rules (VA non-allowable, APR inclusion, tolerance category, financeability) live in crates/mismo/data/fee_rules.json. The file is embedded at compile time as a default but can be replaced at runtime via PUT /api/v1/fee-rules. When VA updates its non-allowable fee list (which happens), the JSON is updated and reloaded — no code change, no deployment required.
Lender credit as negative FeeEntry with 0% tolerance
Lender credits (from premium pricing / rate above par) are stored as positive Cents values and subtracted in total_closing_costs_borrower(). They carry 0% tolerance — the credit cannot decrease from LE to CD (that is itself a TRID violation). The engine validates that credits never exceed total closing costs, which would create impermissible cash-back to the borrower.
Broker comp fractional BPS — dual representation
Broker compensation rates are fractional basis points in real loan files (e.g., "112.76 bps"). LenderCompParsed stores both comp_bps: Option<BasisPoints> (integer, for scenario engine fast-path) and comp_bps_decimal: Option<Decimal> (full precision, for Loan Estimate cent-accurate computation in Epic 11).

Jurisdictional authority

FIPS code is the sole jurisdictional authority
Every fee table lookup, transfer tax calculation, and state-specific rule check keys off CollateralParsed.fips_code. ZIP codes and city names are display-only. "Border properties" (properties whose mailing address appears to be in another state) are handled correctly because the FIPS code is derived from lat/lng geocoding (Epic 5), not from the address string.

CI/CD patterns

macOS runner — rustup run bypasses all proxy issues
After 5 failed approaches (PATH reordering, explicit binary path, RUSTUP_TOOLCHAIN env var, GITHUB_PATH prepend, rustup toolchain install), the definitive fix is rustup run 1.85.0 cargo test .... This bypasses PATH and proxy resolution entirely, directly executing ~/.rustup/toolchains/1.85.0-.../bin/cargo. The Homebrew rustup-init shim that masquerades as cargo is never involved.
Coverage gate: types crate only until epic baselines measured
The 97% coverage gate applies only to the types crate. New crates are added to the exclusion list until their coverage baseline is measured with cargo llvm-cov --package <crate>. The mismo crate has 405 tests covering all public API paths but has uncovered branches (Display impls, error variants) that bring combined coverage below 97%. This will be addressed in a dedicated coverage hardening pass.

Versioning and auditability (Epic 4 design)

DataVersionManifest — perfect replay for regulatory audit
Every analysis stores a DataVersionManifest recording the exact version ID of every data element used: rate sheet, LLPA matrix, MI rate card, fee rules, agency eligibility, lender overlays. Combined with the engine git SHA, any historical analysis can be reproduced exactly. This satisfies CFPB examination requirements (TRID 3-year retention, RESPA 5-year retention).
RefDataStore trait — dev/test/prod storage independence
All reference data access goes through a RefDataStore trait. Three implementations: JsonFileStore (dev/CI, no database), SqliteStore (integration tests), PostgresStore (production). Switching from dev to prod is one configuration line at startup. The business logic in Epics 7–14 never knows which store is in use.
